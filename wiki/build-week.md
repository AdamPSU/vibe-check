How Vibe Check maps to **OpenAI Build Week** (Codex + GPT-5.6 challenge on Devpost).

## Track

**Education** — builders learn accountability for code they wrote (including AI-assisted code), not generic CS tutoring. Sessions stay **repo-concrete**.

## Why it fits the rubric (intent)

| Criterion | How this product answers |
|---|---|
| Quality of idea | Ownership debt from vibe coding; memorable `github` → product path swap |
| Potential impact | Solo builders who must defend real shipped code |
| Design | Clear loop: ground → voice exam → spider |
| Tech implementation | Codex in the **product path** (SDK for grounding / dynamic questions) and as build tool for core logic; GPT-5.6 in grounding synthesis; Realtime for the exam |

## Submission constraints to respect

- Project built with **Codex and GPT-5.6**
- Demo video (<3 min) must **show the product** and **explain use of Codex and GPT-5.6**
- README must describe Codex collaboration
- Provide **`/feedback` Codex Session ID** for the thread where majority of core functionality was built
- Working demo judges can run

## Demo notes (decided)

- Use a **public** repo for the video.
- Grounding phase is **shown** (live calls), then Ready, then ~1 min voice exam, then spider.

## Not decided for submission packaging

Internal split of which human/tool writes which files is intentionally omitted from product docs.
